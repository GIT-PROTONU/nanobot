- **BWT901CL IMU confirmed live (now on /dev/imu → ttyUSB0) @115200** (CH340, fixed baud — NOT 9600). `imu_driver` node parses the WitMotion 0x55 11-byte frames (header/type/4×int16 LE/checksum=sum&0xFF; types 0x51 accel·0x52 gyro·0x53 angle·0x54 mag, no quaternion) and publishes `/imu/data` (sensor_msgs/Imu, orientation derived from Euler), `/imu/mag`, `/imu/euler` (deg) at the device's native ~200 Hz. NOTE: the BWT901CL streams at native ~200 Hz but imu_driver now **throttles publishing to `publish_rate` (default 50 Hz, in robot.yaml)** — cut load1 from ~3 toward ~1.5. The reader still parses every frame (keeps latest values); only publishing is time-gated. imu_driver **CPU was ~26%** (parsing 1000 frames/s at the 200 Hz default, plus a per-byte read spin); cut to **~8%** by (1) batch reads `ser.read(64)` instead of draining `in_waiting` to 1 byte, and (2) sending the WitMotion RRATE command on connect to set the device's stream rate. **Further optimized 2026-06-18** → now **1 Hz=0.5%, 50 Hz (default)=~7-8%, 100 Hz=13%** (was 1 Hz=5%, 100 Hz=20%). Four changes: (a) **device stream rate auto-follows `publish_rate`** — `output_rate_hz` is now `0`=auto (re-sends RRATE on every publish_rate slider change, done in the reader thread so the serial port is only ever touched there); this killed the old ~5% floor where the device streamed 50 Hz while publishing 1 Hz; (b) **pre-allocated/reused** the Imu/Mag/Vector3 messages with covariances set once in __init__; (c) `/imu/euler` (`euler_rate`, default 25) and `/imu/mag` (`mag_rate`, default 10) now publish at their own **low caps** since only /imu/data needs the full rate (the web UI just displays euler; nothing subscribes to mag); (d) precompiled `struct.Struct("<hhhh").unpack_from` straight off the buffer, no per-frame slices; (e) **RSW register** set on connect (`FF AA 02 <bitmask> 00`) so the device emits ONLY accel+gyro+angle (+mag if `mag_rate>0`), dropping the time frame — but **measured: this did NOT lower imu_node CPU** (13.2%→13.3% at 100 Hz), proving the cost at high rate is **rclpy/Python publishing, not parsing**. So the only real lever on imu CPU at a fixed rate is publishing fewer msgs / lower rate (already done via euler/mag caps); to go below ~13%/core at 100 Hz you'd need a compiled node. (f) **reader thread hardened**: `_publish` bails if `not rclpy.ok()`/stopping and the parse loop is wrapped in try/except — fixes a crash where the daemon reader kept calling `publish()` during shutdown ("publisher's context is invalid") and silenced the IMU until a full restart. **KEY MEASUREMENT INSIGHT for the user's "30% CPU" confusion:** `top`/per-process %CPU is **per-core** (one A53 core), but `sys_monitor.cpu_percent` (web UI / OLED) is the **all-4-cores aggregate** — so imu_node at 13% of one core shows as only ~3% in the web UI. The IMU thread runs hot on ONE core (Python publish is single-threaded) while the average looks low. Measure CPU with `/tmp/imucpu.py` style /proc sampler — `pgrep -f imu_node` is BOOBYTRAPPED: it also matches your own measurement script's shell, so filter to the `python` cmdline (`imu_driver/lib/imu_driver/imu_node`), and verify the device serial rate via `/proc/PID/io` rchar (≈publish_rate×55 B/s). imu_driver's reader thread now **(re)opens the serial port with retry** and reconnects if the USB-serial adapter disappears — so the IMU auto-recovers when replugged, no stack restart needed (watch for: USB-A ports are scarce on this board, so adding the webcam can knock the CH340/IMU off `/dev/ttyUSB0` → use a powered hub for both). Both imu_driver and lds_driver_py expose `publish_rate` as a **live-retunable param** — the web UI has sliders that call `/<node>/set_parameters` (rcl_interfaces/SetParameters) directly through rosbridge (no rosapi needed; each node has an `add_on_set_parameters_callback`), and the UI shows each topic's measured Hz. lds_driver_py's publish_rate caps /scan (motor-bound ~5 Hz natural max; lower decimates). - **stack.sh restart is UNRELIABLE at killing nodes** (hit repeatedly 2026-06-18): a restart often leaves the OLD node running (stale 100 Hz imu; stale web on :8080 serving old code → new code 404s). Symptom: two procs for one node, the older one wins the port. Workaround that works: `stack.sh down` → verify with a **python** /proc scan (NOT `pgrep -f`/bash) → `stack.sh up`. **CRITICAL plink gotcha (bit me twice):** `plink -m <localfile>` sends the file's TEXT as the remote shell's argv, so ANY process match (`pkill -f`, `pgrep -f`, OR a python `/proc/*/cmdline` scan) matches the controlling shell itself and kills it (exit 128). FIX: `pscp` the script to the board and run it **by path** (`plink ... "bash /tmp/x.sh"`) so the shell's argv is just the path. Kill stack nodes from a pscp'd python helper run by path.

**Gotchas (now handled by the scripts, but worth knowing):**
- `plink -m <script>` runs the script with its FULL TEXT as the remote shell's argv, so a `pkill -f '<node>'` whose pattern appears in the script **kills the script's own shell** (plink exit 128, truncated output). `stack.sh` runs as a *file* (argv is just the path), so its pkills are safe; only worry about this for inline `plink -m` one-offs (use the bracket trick `oled_display[/]lib` or kill by PID).
- Sourcing `install/setup.bash` under `set -u` dies on unbound `COLCON_TRACE` — wrap with `set +u; source …; set -u` (stack.sh does this).
- Background detached procs with `setsid bash -c "exec …" >log 2>&1 </dev/null &`; in `plink -m` scripts don't `sleep`+`echo` right after a background launch (session can close → exit 128) — check the log in a separate call.
