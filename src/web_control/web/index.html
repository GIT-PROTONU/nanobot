▸ Vision alerts tuning

            

              

                

              

                

              

                

              

                

              

                

              

                

              

                

              

                

              

                

              

                

              

                

              

                

              

                

              

                

              

                

              

                

            

            
Continuous on-GPU frame-diff ("PIR") + optional colour-blob
              tracking + luminance, computed whether or not the video view is open (all
              off when gpu_vision_enable is false). Motion center/
              intercept rate are the motion-saliency and kinetic-intercept Tier-B
              extensions. Motion intercept is the same looming-alert math applied to
              raw motion instead of a calibrated target — no colour needs to be set. Lens
              obstructed flags a flat, dark frame (hand/dust over the lens); try covering
              the camera to see it trigger. Colour cast is the scene's average R/G/B —
              watch it shift under different lighting. Visual interest is how much
              texture/contrast is in frame (try pointing at a blank wall vs. a cluttered
              shelf); overhead structure is the same signal cropped to the top of the
              frame (try holding something up high); shiny surface flags specular
              highlights (try a phone screen or mirror); backlit flags a bright spot in
              an otherwise dim scene; focus/blur flags a sudden sharpness drop (try
              covering the lens partially, close up); motion ↔ target shows how close
              the moving thing is to the tracked colour (needs a target calibrated first).
              Pipeline load is a SOFTWARE-measured proxy — the fraction of each frame's
              time budget spent in the shader+readback block; over 100% means the vision loop
              can't keep up with the configured fps and is quietly falling behind (confirmed on
              real hardware — see the gpu-vision-implemented memory). All have live sliders
              under ▸ Vision alerts tuning — thresholds are tuned from real hardware
              readings where available (obstruction's flatness default), otherwise still
              initial guesses. All informational only so far.
              Optical bumper flags commanded-but-not-moving (possible
              wheel stall/slip) — it reads "clear" almost all the time simply because
              the robot isn't currently being driven; commanded /cmd_vel shows
              exactly what it's seeing so that's visible instead of a single opaque
              flag. To actually see it trigger: drive the robot from the Drive
              tab while something blocks a wheel (or just pick it up) — commanded speed
              stays nonzero, GPU motion stays near zero, and after confirm time
              the bumper should flag. Blob tracking tuning: colour match
              tolerance is how close a pixel must be to the calibrated colour to
              count (higher = more forgiving, catches more shades but more false
              positives); min/max blob size gate how much of the frame must match
              to count as a real lock — raise min if a couple of stray pixels are
              triggering a false lock, lower max if the calibrated colour also
              matches a big wall/background area (use the 🎭 Show tracking mask
              button to see exactly what's matching before tuning these). Picking a new
              target colour resets all three to their defaults; glare derating
              scales the lock confidence down when specular highlights fill the frame,
              so a shiny reflection matching the tracked hue can't hold a false lock
              (0 = off). Named targets is the persistent calibration palette —
              every pick is saved under the name typed in the Camera view, survives
              restarts, and can be re-selected or deleted here.
              Novelty compares the scene to a slow ~20 s background average — it
              stays high while something about the room is genuinely different, then
              habituates; camera frozen flags a capture path that stopped
              delivering (or keeps delivering the identical buffer — a USB/driver
              wedge, not a wheel stall); vibration flags an image much blurrier
              while driving than the standing-still baseline (loose screw / wheel
              imbalance — a maintenance hint); approaching fires when motion is
              growing fast AND centred, i.e. someone is walking up to the robot.
              Several signals now have autonomous consumers in the behaviour layer
              (see robot.yaml behavior:): approach → an anticipatory greeting,
              looming/clutter → caution fast-rules (which throttle speed only if
              slam_nav's trait_motion opt-in is on), novelty → a boosted
              "looking" beat, colour cast → the idle face's ambient mood tint, and the
              scene scalars feed a visual diary folded into reflection.
              Dark reflex: opt-in, auto-toggles the ESP32's
              /led from brightness with hysteresis (on/off thresholds must stay
              apart to avoid flicker).

Navigation (slam_nav)

            

              

            

              

            

              

            

              

            

              

            
            
            
Max speed/turn cap the pursuit controller (also used by
              Motion in the Map view). Stop distance = obstacle range ahead that
              triggers a reactive stop+replan; Robot radius = obstacle inflation used by
              the planner. Stuck timeout aborts a goal if commanded but not actually
              moving for that long ("off" = never). Auto-relocalize is the local
              lost-robot recovery (wide scan-match + spin); Pause SLAM when picked up
              freezes mapping while both wheels are off the ground. All apply live, no
              restart.

            
            

              

            
Parks the lidar spin motor after this long with no motion/goal/
              exploring (0 = never spin down). Wakes instantly on a new goal, exploring,
              teleop motion, pickup, or relocalizing. Untick to disable the whole behaviour
              and keep the lidar always spinning, regardless of the timeout below.

Stress test

            

              statusidle
              workers–
              remaining–
            

            

              

            

              
              
            

            
Pegs every CPU core with low-priority (niced) worker processes
              for the chosen duration — a real-world check that the systemd watchdogs and
              fan curve hold up under load. Workers run at the lowest OS scheduling
              priority, so this page and the rest of the stack keep responding throughout;
              the run auto-stops at the duration regardless (capped at 5 minutes) so a
              forgotten test can't run forever. Watch System health above for live
              CPU%/temp while it runs.
